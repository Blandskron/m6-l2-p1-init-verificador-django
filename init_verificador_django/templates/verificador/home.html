1. Entorno de Ejecución y Herramienta PIP

Aislamiento de componentes e instalación de dependencias en proyectos Django

En el desarrollo profesional con Django, es fundamental evitar la instalación global de paquetes. Por ello, se utilizan entornos virtuales y el administrador de paquetes pip.

¿Qué es un Entorno Virtual?

Un entorno virtual (creado con el módulo venv) es una carpeta aislada que contiene una copia propia del intérprete de Python y su estructura de librerías. Al activarlo, cualquier paquete instalado mediante pip se guardará localmente dentro del entorno y no afectará a otros proyectos de la máquina.

Pasos de Configuración e Instalación del Proyecto:

Consola de Comandos (CMD / PowerShell / Terminal)
# 1. Crear el entorno virtual llamado 'venv' en el directorio del proyecto
python -m venv venv

# 2. Activar el entorno virtual (Comando en Windows)
venv\Scripts\activate

# 2. Activar el entorno virtual (Comando en Linux/macOS)
source venv/bin/activate

# 3. Actualizar el gestor de paquetes pip
python -m pip install --upgrade pip

# 4. Instalar Django mediante PIP (de forma aislada)
pip install django

# 4. Alternativa: Instalar usando el archivo de requerimientos provisto
pip install -r requirements.txt

Comprobación de la instalación:

Para asegurar que Django está instalado correctamente y que se puede invocar desde el entorno activo, existen dos comprobaciones fundamentales:

  • django-admin --version: Comprueba que la herramienta administrativa global del framework está registrada en el PATH del entorno.
  • python -m django --version: Comprueba que el paquete de Python django puede ser importado correctamente por el intérprete actual.

2. Herramientas Administrativas de Django

Diferencia e interacción entre django-admin y manage.py

Django provee dos utilitarios esenciales para interactuar con el framework. Comprender sus roles es vital:

Característica django-admin manage.py
Ámbito Global (del sistema/entorno) Local (específico de este proyecto)
Momento de Uso Antes de crear el proyecto (inicialización) Durante todo el desarrollo del proyecto
Conexión con Settings No conoce el archivo settings.py a menos que se defina la variable de entorno DJANGO_SETTINGS_MODULE. Apunta automáticamente al archivo settings.py de este proyecto.
Ejemplo de uso django-admin startproject mi_proyecto python manage.py runserver

Comandos Administrativos Clave de manage.py:

❖

runserver

Inicia el servidor web de desarrollo local. Por defecto corre en http://127.0.0.1:8000/ pero permite especificar puerto e IP (ej: python manage.py runserver 8080).

❖

startapp (referenciado como "startup" en temarios)

Crea la estructura de directorios y archivos necesaria para un nuevo módulo funcional o aplicación Django (ej: python manage.py startapp verificador).

❖

createsuperuser

Crea una cuenta de usuario administrador con accesos totales para interactuar con la interfaz del panel de administración del framework (/admin/).

❖

migrate y makemigrations

makemigrations analiza los modelos creados y genera los archivos de migración de base de datos. migrate aplica esas instrucciones a la base de datos real (en nuestro caso, SQLite3).

❖

Comandos de Ayuda Django

Usar python manage.py help lista todos los comandos instalados. Usar python manage.py [comando] --help muestra los argumentos detallados de un comando en específico.

3. Estructura del Proyecto y Archivos de Configuración

Mapeo de archivos autogenerados y parámetros configurables de Django

Al generar un proyecto, Django crea un paquete de configuración principal con los siguientes archivos clave:

  • __init__.py: Archivo vacío que le indica a Python que esta carpeta debe ser tratada como un paquete de módulos importable.
  • settings.py: Archivo central con todas las configuraciones del proyecto web.
  • urls.py: Archivo de enrutamiento principal. Mapea expresiones de ruta a vistas.
  • wsgi.py / asgi.py: Puntos de entrada para servidores de producción WSGI (síncronos) y ASGI (asíncronos).

Configuraciones Críticas en settings.py:

A) Configuración de Paths (Rutas de Archivos): Django utiliza la biblioteca pathlib para definir de forma dinámica el directorio raíz (BASE_DIR). Esto previene que se rompan las rutas si el proyecto se mueve de carpeta o servidor.

settings.py (Configuración de Paths)
from pathlib import Path
# BASE_DIR apunta a la carpeta contenedora del proyecto (donde está manage.py)
BASE_DIR = Path(__file__).resolve().parent.parent

B) Configuración de Plantillas (Templates): Para que Django sepa dónde buscar archivos HTML al renderizar vistas, se configura la lista DIRS dentro del bloque TEMPLATES:

settings.py (Configuración de Plantillas)
TEMPLATES = [
    {
        'BACKEND': 'django.template.backends.django.DjangoTemplates',
        # Mapeamos la carpeta 'templates' en la raíz de nuestro proyecto
        'DIRS': [BASE_DIR / 'templates'],
        'APP_DIRS': True, # Permite buscar en la carpeta templates/ de cada app instalada
        # ...
    },
]

Manejo de Espacios de Nombres (Namespaces) en el Enrutador:

Para evitar colisiones entre nombres de rutas de distintas aplicaciones dentro de un mismo proyecto web, se utiliza el Namespacing. En este proyecto se implementa de la siguiente manera:

verificador/urls.py
from django.urls import path
from . import views

# Definimos el namespace de la app
app_name = "verificador"

urlpatterns = [
    path("hola/", views.hola, name="hola"),
    path("", views.home, name="home"),
]

En las plantillas o vistas, se hace referencia a la URL usando la nomenclatura verificador:nombre_ruta, por ejemplo: {% url 'verificador:home' %} o reverse('verificador:hola').

Arquitectura Django: El Patrón MTV (Model-Template-View)

Mapeo del ciclo de Request-Response respecto a MVC tradicional

Aunque Django se describe formalmente como un framework MVC (Modelo-Vista-Controlador), en su terminología interna se aplica el patrón MTV (Modelo-Plantilla-Vista):

M

Model (Modelo)

La capa de acceso a datos. Abstrae la base de datos a través de clases en Python (ORM).

V

View (Vista)

La lógica de negocio. Recibe peticiones HTTP, consulta al Modelo y decide qué renderizar.

T

Template (Plantilla)

La capa de presentación. Archivos HTML con la sintaxis de plantillas de Django para inyectar datos.

Correlación respecto a MVC Tradicional:

  • El Model (Modelo) es idéntico en ambos patrones.
  • La View (Vista) de Django actúa en realidad como el Controlador (Controller) de MVC tradicional.
  • El Template (Plantilla) de Django equivale a la Vista (View) de MVC tradicional.
  • El propio Framework Django actúa como el router/despachador principal que conecta las URLs con la vista correcta.
